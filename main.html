How it works

  1. Submit a report with an optional bounty (wei). The Cigilante engine stores it and returns a report ID.
  2. Reports appear in the list. Unclaimed reports can be claimed by anyone (simulated claimer address).
  3. Stats show total report count, total bounty wei, and number of claimed bounties. Events log recent activity.
  4. Run the Cigilante Java server (e.g. java Cigilante) on port 3952. Set API_BASE below if your server runs elsewhere.

FAQ

What is VigilanteWatch?
VigilanteWatch is the web interface for Cigilante, a community watch bounty ledger. You submit reports (tips), optionally with a bounty; watchers can claim bounties. It runs off-chain; no blockchain required.
Do I need an account?
No. The demo uses placeholder addresses (e.g. 0x0). In production you might tie submissions to wallet or session.
Where does the server run?
By default this page expects the Cigilante HTTP server at http://localhost:3952. Start it with java Cigilante from the Cigilante folder.
What are the limits?
Max report body length 2048 chars, max reports 500, batch query limit 100. Bounty wei is capped by the server.
What error codes can I get?
CG_ReportTooLong, CG_ReportCapReached, CG_BountyOutOfRange, CG_InvalidReportId, CG_ReportNotFound, CG_AlreadyClaimed, CG_BatchTooLarge, CG_InvalidAddress, CG_MissingId.
Is this on a blockchain?
No. Cigilante is an off-chain ledger. Data lives in the Java server process. For persistence you would need to run it against a database or integrate with a chain separately.
What is the Connection section?
It shows the API base URL and the result of the /health endpoint. If status is OK, the Cigilante server is reachable. Use "Check health" to re-run the check.
Does the page auto-refresh?
When you return to the browser tab (window focus), the app runs a debounced refresh of reports, stats, and events so data stays up to date.

API reference

All endpoints expect JSON responses. POST bodies use application/x-www-form-urlencoded.

Glossary

  • Report — A single tip or observation submitted to the watch ledger, with optional bounty wei.
  • Bounty — Optional wei amount attached to a report; claimable by anyone until claimed.
  • Claim — Action that marks a report’s bounty as claimed by a given address (e.g. 0x0 in demo).
  • Watch net — The Cigilante off-chain ledger and its APIs.
  • Chain ref — A reference identifier (e.g. 0x5f2e8a1c9b3d4076) used in docs; not a live chain ID.

Quick start

Run the Cigilante server from the project folder:

cd Cigilante
java Cigilante

Server listens on port 3952. Open this page and use the form to submit a report, then refresh to see it. Use "Unclaimed only" to filter, and click Claim on a report to claim its bounty.

Tips & best practices

  • Keep report bodies under 2048 characters to avoid CG_ReportTooLong.
  • Use a positive bounty wei value if you want the report to be claimable; 0 is allowed.
  • Batch query limit is 100; use offset and limit for pagination when you have many reports.
  • Events are in-memory and limited (e.g. last 200); they are not persisted across server restarts.
  • For production, run Cigilante behind a reverse proxy and set API_BASE in this file to your public URL.
  • Replace placeholder addresses (0x0) with real wallet or session identifiers when integrating with auth.

Error codes

The API returns these error codes in the error field of the JSON body.

  • CG_ReportTooLong — Report body exceeds 2048 characters.
  • CG_ReportCapReached — Maximum number of reports (500) reached.
  • CG_BountyOutOfRange — Bounty wei is negative or above the allowed scale.
  • CG_InvalidReportId — Report ID missing or invalid format (must start with CG-).
  • CG_ReportNotFound — No report with the given ID.
  • CG_AlreadyClaimed — The report’s bounty was already claimed.
  • CG_BatchTooLarge — Requested limit exceeds 100.
  • CG_MissingId — Required id parameter missing (e.g. for /report).

Security notes

Use these guidelines when deploying Cigilante and VigilanteWatch in production.

  • Run the Cigilante server behind HTTPS and a reverse proxy (e.g. nginx, Caddy). Do not expose port 3952 directly to the internet.
  • Validate and sanitize all report bodies on the server; the client may send arbitrary payloads. Cigilante enforces length and format.
  • If you integrate wallet-based auth, verify signatures on the backend before associating reports or claims with addresses.
  • Rate-limit submission and claim endpoints to prevent abuse. The Java server does not include rate limiting by default.
  • Treat bounty wei as off-chain bookkeeping unless you explicitly bridge to a chain; no real funds move in the default setup.
  • Keep dependencies (Java runtime, any libraries) updated. Do not store secrets in client-side code; use env vars or config on the server.

Version & compatibility

VigilanteWatch is the web UI for the Cigilante watch net. This single-file build is self-contained (no build step).

  • API version — Compatible with Cigilante HTTP API as of the bundled server (paths: /reports, /submit, /claim, /stats, /events, /health).
  • Browsers — Tested on recent Chrome, Firefox, Safari, Edge. Requires fetch and basic ES5-style JS.
  • Config — Edit API_BASE in the script section to point to your Cigilante server URL (including port).

Integrations

Ways to use the Cigilante API from outside this UI.

  • cURL — Submit: curl -X POST "http://localhost:3952/submit" -d "body=Hello&bountyWei=0&from=0x0"
  • JavaScript — fetch(API_BASE + '/reports?limit=10').then(r => r.json()).then(console.log)
  • Python — Use requests.get('http://localhost:3952/stats') or requests.post('.../submit', data={...})
  • Wallet links — In a dApp you can pass from and claimer as the connected wallet address after verifying ownership on your backend.

Example requests

Copy-paste examples (replace localhost:3952 if your server is elsewhere).

List reports (first 10):

curl -s "http://localhost:3952/reports?offset=0&limit=10"

Submit a report with bounty 100 wei:

curl -X POST "http://localhost:3952/submit" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "body=Observed activity at 0x123&bountyWei=100&from=0x0"

Claim bounty (reportId from list):

curl -X POST "http://localhost:3952/claim?reportId=CG-xxx&claimer=0x0"

Health check:

curl -s "http://localhost:3952/health"

Troubleshooting

  • Reports not loading — Ensure Cigilante is running (java Cigilante) and listening on port 3952. Check Connection status on this page; if it says Unreachable, fix the server or CORS.
  • Submit fails — Verify body is not empty and under 2048 chars; bounty is a non-negative number. Check the error message shown under the form.
  • Claim fails — Report may already be claimed (CG_AlreadyClaimed). Refresh the list to see current state.
  • Wrong server — Edit API_BASE in this file (script section) to your server URL, e.g. https://your-domain.com (no trailing slash).

Keyboard & accessibility

This UI is form and button driven. Use Tab to move between report body, bounty input, and buttons. Enter in the textarea adds a new line; use the Submit button to send. Buttons are focusable and support focus-visible outline. No custom keyboard shortcuts are defined; screen readers can navigate sections via headings.

About VigilanteWatch

VigilanteWatch is a single-page web interface for the Cigilante community watch bounty ledger. It allows users to submit reports (with optional bounty wei), view and filter reports, claim bounties, and inspect stats and recent events. The backend is the Cigilante Java application, which exposes a simple HTTP API. This front end is static HTML/CSS/JS with no framework; you can host it on any static server and point API_BASE to your Cigilante instance.
